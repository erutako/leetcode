### 思考記録
step2で解く前にstep1での以下の疑問をGeminiで解消してみた。
> 0が連続していなかった場合は、スワップ後の0のインデックスを指しているので、これまた左端の0のインデックスを指していることになる。
> うーん、なんかこの辺がしっくり来てないからすっとコードが書けないんだな。
> で、最初非0が連続していたとして、これは書き込み位置がどんどん右にズレていきつつ、書き込み位置にある値をそのまま書き込んでいるだけなので順番に何も影響しない。
以下ということらしい。
最初はwriteとreadがずっと同じ場所を指し続ける。
0が出てくると、初めて0の領域が生まれてwriteは0の領域の左端を指して、readは次の値を読みに行く。
readが非0を読むと0の領域の左端と非0を交換する。
- 0 ~ write - 1 : すでに確定した非0の領域
- write ~ read - 1 : 0の領域
- read ~ 末尾 : これから調べる領域

いやー、やっぱりよくわかんないな。
不変条件が次に非0を見つけたときの書き込み位置を保持しているというのはわかった。
探索位置はどんどん右にズレていって、非0を見つけたら左端の0とスワップするので、非0はどんどん左に移動していく。
0に遭遇するまではどうするかというと、書き込み位置はどんどん右にずらしたいし、読み込み位置もどんどん右にずらしたい。
違和感は不変条件が非0を見つけたときの書き込み位置の保持なのに、それが最初に0を見つけるまでは非0の値がある位置ということが気になるんだな。

Geminiに聞いて不変条件の解像度を上げる。
なるほどちょっと理解が深まった。
>「write は、（元が0だろうが非ゼロだろうがお構いなしに）左から非ゼロを順番にギュウギュウに詰めていったときの、『現在確定している領域の、すぐ右隣（次の書き込み先）』である」

これはわかるんだけど、最初に0が出るまでは常に自分自身をスワップする動作になるのが気になるんだよな。
それでうまくいくんだけど、なぜうまくいくのか違和感がある。

さらに聞く。
> もし自己スワップを防ごうとすると、人間の頭の中では以下の「2つのフェーズ」を区別することになります。
> フェーズ1（0に出会う前）：「ここは元から正しい位置だから、何もしなくていい」
> フェーズ2（0に出会った後）：「ここは間違った位置だから、正しい位置（write）に移動させなきゃ！」
> しかし、この「2つのフェーズを区別する」こと自体が、実は人間特有の複雑な思考なのです。
> コンピュータ（アルゴリズム）は、フェーズを区別せず、たった1つの冷酷でシンプルなルールで動きます。
> 「非ゼロを見つけたら、問答無用で『今の write の位置』に叩き込む」
> このルールに従うと、0に出会う前の非ゼロ要素に対しても「お前は非ゼロだな！よし、今の write の位置（＝自分がいま居る場所）に叩き込んでやる！」と愚直に処理します。
> 結果として自己スワップが発生しますが、これは「意味のない動作」というより、**「『君の正しい居場所はここだよ』と、同じ場所に再配置している動作」**と捉えるのが、アルゴリズムの世界の解釈です。

不変条件を見つけて、それを満たすコードを書いて、それを満たし続けるなら別に問題ないと判断するという感じかな。
もし非0しかないような極端な入力が与えられても不変条件を満たすのであればそれは問題ないと。
それはそう。

なんかわかってきたぞ。
Gemini曰く、これは0を右に集めるのではなく非0が必ず左側に集まっているといえると。
確かに。
そして、初期状態・ループ中・最終で不変条件である探索位置より左は0であるということを守り続ければ解答になる。
それに則って考えると非0が見つかったらそのままでいいんだけど、0が見つかったら次の非0とスワップしたいということになる。
だから非0確定区間の次の0を覚えておいて、非0が見つかったらそこと交換したい。
最初に0が見つかるまでも非0確定区間はどんどん右にズレていって、その場所に見つかった非0を置きたい。
これでだいぶスッキリした！

コード書く。
3minぐらいでかけた。
でも、非ゼロのときにread, writeでswapするところがまだ感覚的に違和感があったな。
writeはいいんだよな。非0確定区域後の位置だから、そこにreadしたやつをいれるのは納得。
非0確定区域後の値をreadした位置に入れるのはなにか違和感。
readはなにの位置かというと、今探索している位置なんよね。
探索位置の最後に0を運ぶことは納得。
これが非0のときは常に自己スワップになるから問題ないんだけど、やはりここがひっかかるんだよな。

最終的にだいぶ腑に落ちた。
不変条件は左側が非0のみの区間になっていること。
最初はwriteが0から始まっていて、非0が見つかるたびに書き込み位置が右にずれる。
0が見つかったときはそのまま右にずらすと非0確定区間に0が含まれることになるので、立ち止まる。
次に非0が見つかると非0確定区間の最後に入れてあげる。
このときスワップ対象の値はとにかく非0確定区間より右側に入れておけば不変条件は守られる。
今回のコードでは探索位置の最後に入れられており、この不変条件が守られる。
また、実際にシミュレーションしてみると、スワップするときのreadは必ず非0でwriteは0もしくはreadしている非0自身になる。
なぜかというと書き込み位置は0にあたった時点でその位置の値になり、0にあたるまではreadしている非0自身の位置になるから。
まだすごいしっくりまではいかないけど、コード書くにあたってすっとかけるくらいの理解度にはなった。

### コード
```ruby
# @param (Integer[]) nums
# @return {Void} Do not return anything, modify nums in-place instead.
def move_zeroes(nums)
    write = 0
    nums.each_index do |read|
        next if nums[read].zero?
        nums[write], nums[read] = nums[read], nums[write]
        write +=1
    end
end
```
