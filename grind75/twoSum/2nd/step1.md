最近仕使っているRubyで解き直す

### 課題の言語化(できたら現実世界で同じ課題を再現する)
足すとtargetになる2つの数字を見つける。
数字が書かれたカードが順番に並んでいる。
1枚目のカードから順番にtargetと引き算をする。
引き算した答えの数字が書かれた箱にカードを置く。
以降、以下の手順を繰り返す。
1. そのカードの数字が書かれた箱にカードが入っているか確認する。入っていればそれが答えの組み合わせ。
2. 入っていなければtargetと引き算する。引き算した答えの数字にそのカードの順番の数字を入れる。

### 思考記録
O(N)でやれそうという直感。
組み合わせはハッシュを使えば計算量が少なそうという感覚。

### コード
```ruby
# @param {Integer[]} nums
# @param {Integer} target
# @return {Integer[]}
def two_sum(nums, target)
    hash = {}
    nums.each_with_index do |num, index|
        return [hash[num], index] if hash.key?(num)
        hash[target - num] = index
    end
end
```
### 感想(他の方のコードなどを読んで)
- 問題の設定上あり得ないが、答えが見つからなかったときにどうするか考えたほうが良さそう。
  - 空配列かnil？
  - Rubyあんまり知らないけど、Rubyならnilを返したくなる。とりあえず答えが存在しなかった感がnilのほうが出る気がする。
- 引数に渡された配列の要素数が1以下のケースを考えることもできるのか
  - 後置ifで値が返されないときはnilを返すと、異常系の結果であることは伝わってよさそう
  - 引数に原因があることを知りたい場合は最初にvalidationして例外をraiseするかぁ
    - 引数のvalidationに引っかかったか、答えが見つからなかったかなど、どのような理由で問題が起きたか呼び出し元が把握したい場合はケースごとに例外をraiseすることになりそう。
      - 理由を把握した上で処理を継続したい場合は呼び出し元で握りつぶしてもらう
- LeetCodeのテスト入力では存在しないけど、同じ値のペアがtargetになるケースを明示的にスキップ書いてもいいかも。そのほうが仕様がわかりやすい。
- どういうテストを書きたくという視点だと色々ケアしたいケースが見つかりそう。(個人的に)
- Brute Forceだと空間計算量がO(1)なのか、場合によっては選択肢になりそう
- One-PassとTwo-Passだと、Two-Passを選ぶメリットはなさそう
  - 一応整数倍ではあるがOne-Passのほうが計算量少ない
  - メモリ効率もearly returnする可能性がある分、One-Passのほうが良くなる可能性がある
